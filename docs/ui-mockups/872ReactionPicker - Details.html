Description

A UI control that allows a user to select from a set of reactions (e.g., Like, Celebrate). It presents a primary action and reveals more options on hover or focus, adapting for touch devices.

Design System Alignment

Utilizes design system tokens for colors, spacing, shadows, and typography. Composed of atomic Button, Icon, and Tooltip components.

Responsive Specifications

Mobile Adaptations

  • On touch devices, the picker is designed to be triggered on a long-press (simulated via class change in this static mockup).
  • Ensures all reaction buttons maintain a minimum touch target size of 44x44px.

Tablet Adaptations

  • Functions identically to the desktop version, responding to hover or touch.

Desktop Enhancements

  • The reaction panel expands on hover over the trigger button for quick interaction.
  • Tooltips appear on hover over individual reaction icons.

Accessibility Features

Keyboard Support

The main trigger button is focusable. When the panel is open, users can `Tab` through the reaction buttons. `Enter`/`Space` activates a reaction.

Screen Reader Support

The main trigger has an `aria-label` like 'React to post'. Each reaction button has a specific `aria-label` (e.g., 'Like', 'Celebrate'). The panel container uses `role='toolbar'`.

Focus Management

Using `:focus-within` allows the panel to remain open while the user tabs through the inner buttons. A JavaScript implementation would be required to trap focus.

Aria Attributes

  • aria-label
  • role="toolbar"
  • aria-haspopup="true"
  • aria-expanded="false"

Color Contrast

All icons and text will meet WCAG 2.1 AA contrast ratios.

Implementation Notes

Embedded Css Architecture

All styles are embedded. CSS uses BEM for the `reaction-picker` block. Design system tokens are used for consistency.

Component Organization

The HTML showcases the component in its default (collapsed) and expanded states. A CSS-only `:hover` and `:focus-within` mechanism is used to demonstrate interactivity.

Responsive Design

The component is intrinsically responsive. On touch devices, a tap-and-hold (simulated by adding a class) would be the intended trigger, while desktop uses `:hover`.

Accessibility Compliance

ARIA roles and labels are used to provide context to assistive technologies. Keyboard navigation is supported through standard button behavior.

Browser Support

Uses modern CSS supported by all major evergreen browsers.

Usage Guidelines

When To Use

Used within a `PostCard` or similar content item to allow user engagement.

Content Guidelines

Icons should be universally recognizable emojis or symbols for reactions.

Placement Rules

Typically placed in the action bar at the bottom of a content card.

Customization Options

  • Add `data-selected="true"` to the trigger button to show an 'active' state, along with changing the icon and text to reflect the selected reaction.

Testing Recommendations

Browser Testing

Test in Chrome, Firefox, Safari, Edge

Device Testing

Test hover on desktop. Test tap/long-press behavior on mobile/tablet.

Accessibility Testing

  • Keyboard navigation through all options.
  • Screen reader announcement of the toolbar and each button's label.
  • Color contrast of selected state.

Performance Testing

Animations should be smooth and not cause layout shift.
